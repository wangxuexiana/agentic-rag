总流程

当前主链路仍然是“先规划、再确认实体、再多路检索、再证据治理、最后回答”的 Agentic RAG。新加的能力不是平行替代主链，而是嵌进原有节点中：查询侧多层缓存负责短路重复计算，导入侧 chunk 动态更新负责让本地召回底座持续保持最新。

总原则：节点之间不是传统“上一个函数 return 给下一个函数参数”，而是都围绕同一个 QueryGraphState 工作。真正的数据流就是 state 字段在各节点之间不断增量变化。

1. 入口层

app/query_process/api/query_service.py

组装初始 state，写入用户问题、会话标识、默认轮次和历史信息，然后启动 LangGraph。

2. 规划层

node_planner → node_item_name_confirm → node_tool_router

先判断问题类型、是否需要澄清、是否要查本地/联网/图谱，再翻译成执行开关。planner cache

3. 检索执行层

embedding / hyde / web_search / kg

按路由开关并发或按需执行多路召回，生成本地 chunk、HyDE 候选、网页结果和图谱占位结果。retrieval cache

4. 融合精排层

node_rrf → node_rerank

先融合本地多路召回，再把本地和 web 证据统一重排，形成最终证据池。rerank cache

5. 证据治理层

node_evidence_reflection → node_dynamic_reretrieval

判断证据是否足够支撑回答；如果不够，则重写 query、升级工具、再补检一轮。

6. 输出层

node_answer_output

决定是直接返回澄清、保守回答，还是正式生成答案，并完成来源、图片、历史落库和流式收口。answer cache

7. Chunk 动态更新

node_diff_chunks → node_bge_embedding → node_import_milvus

导入侧按 chunk_key / chunk_hash 做 diff，只重算新增/更新 chunk，只删除被删 chunk，使查询侧召回永远建立在最新分片之上。incremental import

主图分支

分支 1：商品确认后是否直接返回

如果 node_item_name_confirm 已经产生了澄清问题或候选确认问题，后续不会继续检索，而是直接进入 node_answer_output。

典型场景：只有型号没有诉求、多个商品候选冲突、Planner 先要求澄清。

分支 2：Reflection 后是否补检索

如果 evidence_status 是 insufficient 或 conflicting，且轮次没超限，会进入 node_dynamic_reretrieval，重写 query 并调整工具选择。

分支 3：缓存是否短路节点执行

Planner、检索、Rerank、Answer 四层都可能被缓存命中。命中后不会改变图结构，但会跳过该节点的重计算，直接写回结果字段。

Query Cache

缓存不是单独一条链，而是分散在原主链多个节点中的短路层。命中时节点仍然“走到”，但会跳过重算并直接回写结果字段。

缓存覆盖阶段

  • planner cache：复用意图识别、任务类型、工具规划与澄清结论。
  • retrieval cache：复用 embedding / HyDE 召回结果与 hyde_doc。
  • rerank cache：复用统一候选集的精排结果 reranked_docs。
  • answer cache：复用最终答案文本、部分来源组织结果和构造后的 prompt。

缓存键稳定化

  • 通过 resolve_stable_query_text() 尽量把口语化 query 归一为稳定 key。
  • planner / answer key 会带上 prompt 版本或模型相关因素，避免旧结果污染新逻辑。
  • retrieval key 会编码 retrieval_type、topk、索引版本等参数。
  • rerank / answer key 需要包含候选文档身份，否则无法保证语义等价。

Chunk Sync

这部分不在查询图内部执行，但它直接决定查询图中本地召回能拿到什么内容，因此必须并入总体说明。

增量更新主流程

  1. 文档切分后得到当前版本 chunk 集合。
  2. 从 document_registry 读取历史 snapshot。
  3. 按 chunk_key 和 chunk_hash 比较，得到 added / updated / deleted / unchanged。
  4. 只对 added + updated 重做 embedding。
  5. 对 deleted 和旧版本变更 chunk 做定向删除与 metadata 同步。
  6. 回写最新 snapshot，供下一次 diff 使用。

对查询链的影响

  • 查询节点无需感知 diff 细节，但本地召回底层内容是动态更新后的最新 chunk。
  • 比全量重建更快，也减少了旧 chunk 残留、章节迁移后重复命中的问题。
  • 如果缓存 key 没绑定索引版本，理论上可能复用到旧召回结果，因此 retrieval cache 设计必须考虑版本因素。

Planner

app/query_process/agent/nodes/node_planner.py

输入

  • original_query
  • history

做什么

  • 判断是不是参数问答、比较题、泛问、时效性问题。
  • 决定是否先澄清。
  • 决定首轮是否启用本地库、HyDE、Web、KG。

输出

  • intent_type
  • task_type
  • selected_tools
  • need_clarify
新增点：节点入口先走 get_planner_cache()，规划完成后用 set_planner_cache() 回写；命中时仍会把结果回填到 state，所以下游节点不需要知道它来自缓存还是实时规划。

实体确认

app/query_process/agent/nodes/node_item_name_confirm.py

输入

  • original_query
  • history
  • 商品名候选库

做什么

  • 抽取或补全 item_names。
  • 生成更适合检索的 rewritten_query。
  • 对齐商品库并决定是确认、候选澄清还是无实体继续检索。

输出

  • item_names
  • rewritten_query
  • confirmation_mode
  • 必要时直接写 answer

Tool Router

app/query_process/agent/nodes/node_tool_router.py

节点职责

把 Planner 给出的高层工具计划翻译成图内各检索节点可执行的布尔开关，例如 run_embedding、run_hyde、run_web_search、run_kg。

为什么单独有这一层

这样图结构可以保持稳定，具体某一路是否执行由状态控制，而不是运行时动态改图。二轮补检索时也只需要重写工具开关即可。

多路检索

Embedding

app/query_process/agent/nodes/node_search_embedding.py
  • 核心输出：embedding_chunks
  • 支持 item_name 过滤与全库召回兜底
  • 命中 retrieval cache 时跳过向量生成和 Milvus 查询

HyDE

app/query_process/agent/nodes/node_search_embedding_hyde.py
  • 先生成 hyde_doc，再做二次向量召回
  • 用于弥补短问句、弱表达和隐含意图问题
  • 同样支持 retrieval cache

Web Search

app/query_process/agent/nodes/node_web_search_mcp.py
  • 输出：web_search_docs
  • 补充时效性、官网和下载类信息
  • 后续在 rerank 阶段与本地文档统一打分

KG

app/query_process/agent/nodes/node_query_kg.py
  • 当前更多是占位支路
  • 为后续图谱问答和关系推理预留入口
  • 状态字段统一为 kg_chunks

RRF

app/query_process/agent/nodes/node_rrf.py

这一层只负责本地多路结果融合，主要处理 embedding_chunks 和 hyde_embedding_chunks 的去重、对齐和 Reciprocal Rank Fusion 计算，输出 rrf_chunks。Web 结果不在这里和本地结果混排，而是留到下一层统一标准化后再进入 Rerank。

Rerank

app/query_process/agent/nodes/node_rerank.py

输入

  • rrf_chunks
  • web_search_docs
  • rewritten_query

做什么

  • 统一本地和 web 的文档结构
  • 调用 reranker 计算相关性
  • 动态截断 TopK，产出最终证据池

输出

  • reranked_docs
  • 供 reflection 和 answer 共享使用
新增点：这里加入了 get_rerank_cache() / set_rerank_cache()。同一候选文档集合二次进入时，可直接复用精排结果，避免反复 rerank。

Evidence Reflection

app/query_process/agent/nodes/node_evidence_reflection.py

输入与判断

输入是 rewritten_query、item_names、reranked_docs。节点会判断现有证据是否足够支持回答，还是存在缺失、冲突、空洞。

输出与控制

  • evidence_status：例如 sufficient、insufficient、conflicting、unknown
  • missing_facts：指导下一轮补检索
  • confidence：辅助 answer_output 做输出策略

Dynamic Re-retrieval

app/query_process/agent/nodes/node_dynamic_reretrieval.py

节点职责

当 Reflection 认为证据不足且轮次允许时，生成新的 followup_query、升级或收缩 selected_tools，并把 retrieval_round 加一，回到 ToolRouter 再来一轮。

为什么重要

这一步把系统从“固定召回一次”提升为“基于证据反思的可控二轮检索”，是当前链路区别于普通 RAG 的关键点之一。

Answer Output

app/query_process/agent/nodes/node_answer_output.py

进入条件

  • item_name_confirm 已经给出澄清问题
  • reflection 判断证据已足够
  • 或补检索结束后进入最终收口

节点动作

  • 优先使用上游已写入的 answer
  • 否则查 answer cache
  • 未命中时正式构造 prompt 生成答案
  • 整理来源、图片、done_list 和历史消息

输出

  • answer
  • sources
  • done_list
  • cache_stats
