# 离线评测迭代时间线报告

## 1. 这份报告解决什么问题

这份报告的目标不是单纯罗列 `test/offline_eval` 里的文件，而是把这些评测产物串成一条可以对面试官讲清楚的迭代时间线：

- 每一轮主要改了什么能力
- 为什么这些改动会影响召回率、准确率、通过率
- 哪些指标是真正的主链质量提升
- 哪些指标变化其实是评测层增强，而不是业务主链变强

结论先说：

1. 这个项目的优化脉络非常清晰，先修“链路能不能跑通”，再修“Planner 和澄清策略”，最后才进入“答案层和评测层治理”。
2. 真正能代表主链质量变化的核心报告，主要是 `seed50_report.json` 和 `2026-05-17` 的 `score` 报告。
3. `ragas` 报告可以讲评测体系成熟度，但不能把 `faithfulness` 的提升全部讲成业务问答质量提升，因为其中一部分来自评测脚本本身的答案清洗和重试增强。

## 4. 迭代时间线

## 4.1 阶段 0：先把离线评测体系跑通

对应文件：

- `sample_report.json`
- `mcp_sample_report.json`
- `local_eval_smoke.report.json`
- `local_eval_smoke2.report.json`
- `cli_smoke.report.json`

这一阶段新增或验证的能力：

- 最小离线评测 case 跑通
- MCP 版本的样例评测跑通
- 本地服务替代版本评测跑通
- CLI 统一入口补齐
- `list-failed / diff / summarize / rerun-failed / trace / trends` 这类评测运维能力补齐

从面试表达上，这一阶段你可以这样讲：

> 我不是直接上来调问答效果，而是先把离线评测基础设施搭起来。这样后面每次改 Planner、检索策略、答案输出，都能有稳定的回归指标，而不是靠人工感觉判断。

这一阶段的评测分数都是 `1/1` 或 `3/3`，它们的意义不在于“模型多强”，而在于“评测体系已经具备可用性”。


## 4.2 阶段 1：第一版主链优化，先解决链路是否跑通

这部分最关键的信息在 [第一版优化总结.md](../第一版优化总结.md) 里已经写得很明确。

第一版优化前后，对 `seed50` 的核心指标变化是：

- `pass_rate: 0.22 -> 0.36`
- `passed: 11/50 -> 18/50`
- `tool_selection_accuracy: 0.68 -> 0.98`
- `answer_mode_accuracy: 0.46 -> 0.66`
- `must_have_accuracy: 0.70 -> 0.80`
- `abstain_accuracy: 0.88 -> 1.00`
- `evidence_status_accuracy: 0.84 -> 0.90`
- `avg_final_confidence: 0.3296 -> 0.4626`

这一版改了什么：

1. 商品确认节点不再对“单个中置信候选”过度澄清  
   代码位于 [node_item_name_confirm.py](../app/query_process/agent/nodes/node_item_name_confirm.py)

2. 本地知识库问答增加默认兜底召回策略  
   第一轮至少走 `embedding + hyde`，避免单路召回过于保守  
   代码位于：
   - [node_planner.py](../app/query_process/agent/nodes/node_planner.py)
   - [node_tool_router.py](../app/query_process/agent/nodes/node_tool_router.py)
   - [node_dynamic_retrieval.py](../app/query_process/agent/nodes/node_dynamic_retrieval.py)

3. 低置信度答案保护  
   当 `evidence_status` 不足且置信度低时，不允许模型强答，转保守回答或拒答  
   代码位于 [node_answer_output.py](../app/query_process/agent/nodes/node_answer_output.py)

这一版为什么能提分：

- 商品确认不再误伤，直接提升了操作类和参数类问题的可达性
- 双路召回让本地知识库问题不再只靠单条召回结果吃饭
- 低置信保护减少了“证据不足还硬答”的错误

这一版提升最明显的类别：

- `ready_local_operation: 0.0 -> 0.6667`
- `ready_abstain: 0.3333 -> 0.6667`
- `ready_local_param: 0.6 -> 0.7`
- `doc_coverage_seed: 0.2 -> 0.3`

面试里这一版最适合讲成：

> 第一版优化我主要先解决链路能不能稳定跑通的问题，重点修了商品确认过度保守、检索默认兜底不足、以及证据不足时模型还会强答的问题。结果是工具选择准确率从 68% 直接拉到 98%，整体通过率从 22% 提升到 36%。


## 4.3 阶段 2：第二版主链优化，重心从“修链路”转到“修 Planner 策略”

这部分同样在 [第一版优化总结.md](../第一版优化总结.md) 里有完整记录。

第二版相对第一版的指标变化：

- `pass_rate: 0.36 -> 0.48`
- `passed: 18/50 -> 24/50`
- `task_type_accuracy: 0.68 -> 0.78`
- `clarify_accuracy: 0.90 -> 0.94`
- `answer_mode_accuracy: 0.66 -> 0.72`
- `second_round_rescue_accuracy: 0.8571 -> 0.9286`
- `must_have_accuracy: 0.80 -> 0.86`
- `avg_final_confidence: 0.4626 -> 0.5114`

这一版主要改了什么：

1. Planner 对“泛问题”和“对象不明确问题”更倾向先澄清
2. 比较类问题的路由更保守，避免误挂到 `kg` 或错误对象
3. 二轮补检索触发条件和 follow-up query 更稳

这一版为什么能提分：

- 该澄清的问题不再被直接强答
- 比较题不再一开始就路由偏
- 二轮补检索不再只是“触发了”，而是更可能“触发后有用”

这一版提升最明显的类别：

- `clarification pass_rate: 0.125 -> 0.375`
- `clarification.task_type_accuracy: 0.375 -> 0.75`
- `comparison_seed pass_rate: 0.0 -> 0.6667`
- `comparison_seed.task_type_accuracy: 0.0 -> 1.0`
- `ready_second_round_web pass_rate: 0.25 -> 0.5`

面试里这一版最适合讲成：

> 第二版优化我把重心从商品确认转到了 Planner。因为第一版之后，剩余错误开始集中在两类：一类是泛问题该澄清却被直接回答，另一类是比较题路由过激。第二版我重点修的是前置任务规划，所以通过率又从 36% 提升到了 48%，比较题通过率从 0 提升到 66.67%。

## 4.5 阶段 4：2026-05-17 这轮主链结果，体现的是“入口继续变准，但回答层暴露新问题”

对应文件：

- [offline_eval_cases.seed50.run.20260517_125927.score.20260517_131555.json](./offline_eval_cases.seed50.run.20260517_125927.score.20260517_131555.json)

它相对 `seed50_report.json` 的变化是：

- `pass_rate: 0.54 -> 0.50`
- `intent_accuracy: 0.86 -> 0.88`
- `evidence_status_accuracy: 0.84 -> 0.88`
- `answer_mode_accuracy: 0.70 -> 0.60`
- `avg_final_confidence: 0.3416 -> 0.371`
- `second_round_trigger_rate: 0.56 -> 0.58`

按类别看，变化最明显的是：

- `ready_abstain: 0.6667 -> 1.0`
- `doc_coverage_seed: 0.3 -> 0.4`
- `clarification: 0.5 -> 0.0`

这个现象很有价值，因为它能体现你对系统问题的理解深度：

1. 前置判断并没有变差，反而更准了  
   `intent_accuracy` 和 `evidence_status_accuracy` 都提升了

2. 但最终回答模式变差了  
   `answer_mode_accuracy` 从 `0.70` 掉到 `0.60`

3. 也就是说，问题已经不在“前面不知道该怎么规划”，而在“后面明明识别对了，却没有把它稳定落成正确输出”

对面试官，这一轮可以这样讲：

> 后面这轮离线评测很有意思，它不是单纯提分或者掉分，而是暴露了系统演化阶段的变化。前面的意图识别和证据状态判断更准了，但 `clarification` 类别的最终输出掉了，说明问题已经从“前置路由失真”转成了“回答层和澄清落地不稳定”。这类现象对我来说是有价值的，因为它告诉我下一轮优化该往哪里打，而不是盲目继续调检索。

这里要特别注意：

- 这部分主要体现的是“评测体系增强”
- 不能简单把所有分数上涨都讲成“问答主链更强了”

### 5.1 初始接入 `ragas`

初始 `ragas` 报告：

- `faithfulness_count = 11`
- `faithfulness_mean = 0.5273`
- `response_relevancy_count = 50`
- `response_relevancy_mean = 0.3168`
- `context_recall_count = 1`
- `context_precision_count = 10`

暴露的问题：

- 评测答案尾部混入了 `参考说明 / 来源 / 置信度`
- 一部分指标可算，一部分指标因连接问题几乎算不出来

### 5.2 清洗答案后

清洗答案尾部后：

- `faithfulness_mean: 0.5273 -> 0.8159`

但这一步不能讲成“主链回答质量突然暴涨”，更准确的说法是：

> 评测口径更合理了。之前 `ragas` 把答案后面的解释性尾巴也算进去了，导致忠实度被低估。清洗后，评测更接近用户真正关心的主答案部分。

### 5.3 加入重试退避后

重试增强后的 `ragas` 报告：

- `faithfulness_count: 11 -> 24`
- `faithfulness_mean = 0.8331`
- `response_relevancy_count: 50 -> 32`
- `response_relevancy_mean = 0.338`
- `context_recall_count: 1 -> 10`
- `context_recall_mean = 0.7`
- `context_precision_count = 10`
- `context_precision_mean = 0.7`

这一轮真正说明的是：

- 你已经把 `ragas` 从“偶尔能跑”变成“能比较稳定地产出结果”
- `faithfulness` 和 `context_recall` 的覆盖率明显变高
- 评测脚本具备了抗连接抖动能力

但仍存在限制：

- `403 AllocationQuota.FreeTierOnly`
- `response_relevancy` 仍然偏低

面试时这部分建议这样讲：

> 后来我又把离线评测从规则评测扩展到了 `ragas`。我不是简单把库装上，而是补了 `retrieved_contexts`、答案清洗、重试退避和总报告合并。这样我不仅能看工具路由和澄清对不对，还能看 `faithfulness / context_recall / context_precision` 这类更接近 RAG 质量本身的指标。


## 5. 面试时最推荐的讲法

如果面试官问“你到底做了哪些迭代，怎么证明有效”，最推荐的回答结构是下面这四段。

### 5.1 先讲评测体系

> 我先把离线评测体系搭起来了，不只是跑样例，还补了本地版、MCP 版和 CLI，能做 failed case 回放、report diff、趋势汇总。这样后面的每次优化都不是拍脑袋，而是能看离线指标变化。

### 5.2 再讲第一版优化

> 第一版我主要修链路稳定性，重点是商品确认别过度保守、本地问答默认走双路召回，以及证据不足时别硬答。结果是 `seed50` 的通过率从 22% 提升到 36%，工具选择准确率从 68% 提升到 98%。

### 5.3 再讲第二版优化

> 第二版我把重点转到 Planner，解决泛问题该澄清却不澄清、比较题路由过激的问题。结果通过率从 36% 提升到 48%，`clarification` 类通过率从 12.5% 提升到 37.5%，`comparison_seed` 从 0 提升到 66.67%。

### 5.4 最后讲当前阶段认识

> 到后面的版本，前置路由已经比较准了，主问题开始转移到答案层控制、二轮 Web 证据质量和文档覆盖完整性上。我又把 `ragas` 接到离线评测层，这样不仅能看流程对不对，还能看答案忠实度和上下文召回质量。


## 6. 一句话版总结

如果只让你讲一句话，可以这么说：

> 这个项目的优化路径是先把 Agentic RAG 主链从“容易跑偏”修到“能稳定跑通”，再把 Planner 和澄清策略从“粗放路由”修到“更贴近问题类型”，最后再把评测体系升级到规则评测加 `ragas` 双层评估，所以我不仅知道系统哪次提分了，也知道每次提分究竟是前置路由、检索补救、还是答案控制带来的。
