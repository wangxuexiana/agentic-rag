## 2. 当前已知问题

### 2.1 `response_relevancy` 仍然偏低

当前 `response_relevancy_mean = 0.338`，明显低于默认阈值 `0.75`。

这通常意味着下面至少有一类问题仍然存在：

- 某些 case 的最终回答没有正面回答问题
- 某些 case 的回答里仍带了额外说明，影响纯答案相关性
- 某些离线 case 的 `gold_answer` 或提问方式本身不够聚焦

这个问题目前还没有被修复，只是被暴露得更清楚了。


### 2.2 `ragas` 受评测模型可用性影响很大

`ragas` 不是纯本地打分，它依赖评测模型调用。

本次真实运行中，已经出现过两类外部问题：

- `Connection error.`
- `403 AllocationQuota.FreeTierOnly`

前者通过脚本里的重试退避已经缓解；后者属于评测模型额度问题，脚本本身无法解决。

结论：

- 只要评测模型额度不足，`faithfulness` 和 `response_relevancy` 的覆盖率就会下降
- 看到分数为空时，要先查 `metric_errors`，不要直接把空值理解成“分数很低”


### 2.3 Windows 控制台日志可能乱码

在 Windows `gbk` 控制台下，`loguru` 输出特殊字符时可能出现乱码或编码告警。

当前影响范围：

- 影响终端展示
- 不影响 `json/jsonl` 报告文件写入

因此遇到日志显示异常时，优先检查落盘报告，不要只看终端输出。


### 2.4 当前基线不是“永久最新”

当前基线报告只代表当时那次：

- 离线 case 数据
- 问答主链实现
- `ragas` 评测脚本版本
- 评测模型可用性

其中任一项变化后，这组结果都不应该继续被当成“当前最新基线”。


## 3. 当前最值得推进的事

如果短期内不继续跑大批量评测，优先级最高的是这三件事：

### 3.1 整理 case 质量

优先检查：

- `gold_answer` 是否足够明确
- `expected_evidence_status` 是否稳定可判
- `must_have` / `must_not_have` 是否存在歧义

case 质量差，会直接把规则评测和 `ragas` 一起拉偏。


### 3.2 追踪低 `response_relevancy` case

建议直接从合并报告里筛：

- `all_case_scores[].ragas.response_relevancy < 0.75`

重点看这些 case 是否属于：

- 回答跑题
- 过度模板化
- 该拒答却回答了
- 该直答却加了太多尾部说明


### 3.3 固定一份小样本 smoke 集

目前 `seed50` 适合回归，但每次都全量跑较慢。

建议后续补一份 `5-10` 条的小样本：

- 覆盖本地参数问答
- 覆盖操作步骤问答
- 覆盖二轮检索
- 覆盖澄清
- 覆盖拒答

这样后续换评测模型、调 `ragas` 逻辑时，可以先低成本验证。
